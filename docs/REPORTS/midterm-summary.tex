\section{项目概况}

Snooker2D 是一个基于 C++17 和 Qt 6 的 2D 斯诺克台球游戏，采用课程讲授的 MVVM 五层架构（Common / Model / ViewModel / View / App）。项目三人协作，10 天工期，当前处于第 5 天中期检查节点。

\section{技术架构}

项目严格按照 MVVM 模式拆分为五个独立编译的静态库：

\begin{center}
\begin{tabular}{|c|l|l|l|}
\hline
\textbf{层} & \textbf{职责} & \textbf{依赖} & \textbf{负责人} \\
\hline
Common & 全局类型、常量、数学工具 & 无 & C \\
Model & 游戏状态机、物理引擎、规则判定 & Common + Qt::Core & B \\
ViewModel & 数据转换、属性暴露、命令绑定 & Model + Qt::Core & B \\
View & 界面渲染、用户输入 & ViewModel + Qt::Widgets & A \\
App & 对象装配、依赖注入 & 所有层 & C \\
\hline
\end{tabular}
\end{center}

数据流方向为 Model（信号）→ ViewModel（属性）→ View（重绘），反向为 View（用户操作）→ ViewModel（命令）→ Model（状态变更）。Model 层不依赖 Qt Widgets，View 层不直接访问 Model。

\section{架构落地情况}

\begin{enumerate}[leftmargin=*]
\item 五层 target 在 CMakeLists.txt 中编译为独立静态库，自底向上链接
\item Model 层纯 C++，不依赖 Qt Widgets，通过 QObject 信号与 ViewModel 通信
\item ViewModel 通过 Q\_PROPERTY 暴露数据、Q\_INVOKABLE 暴露命令
\item View 通过 setViewModel() 接收 ViewModel，不直接接触 Model
\item App 层在 MainWindow 中完成全部对象的依赖注入和跨层信号绑定
\end{enumerate}

\section{技术难点的克服}

\subsection{vcpkg 与 MinGW 的兼容性问题（C）}

项目初始配置时，vcpkg 的 MinGW triplet（x64-mingw-dynamic）无法找到预编译的 Qt 6 二进制包，触发从源码编译 Qt 的流程，预计耗时数小时。经排查，vcpkg 内部使用独立的 MinGW 工具链查找机制（通过 find\_program 搜索 x86\_64-w64-mingw32-gcc），与 MinGW-Builds 独立包的命名约定不完全匹配。

\textbf{解决方案}：放弃 vcpkg 管理 Qt，改用 aqtinstall 直接下载 Qt 官方预编译的 MinGW 版本（6.10.3）。CMake 通过 CMAKE\_PREFIX\_PATH 以 find\_package 方式解析 Qt，绕过 vcpkg 的 toolchain 机制。同时创建编译器别名文件（x86\_64-w64-mingw32-gcc.exe），确保 vcpkg 的工具链查找机制可正常工作。

\subsection{物理引擎参数调优（B / C）}

物理引擎需要三个关键参数：球-球碰撞恢复系数、库边反弹系数和摩擦力系数。初始值（0.96 / 无 / 0.98）与实际台球运动偏差较大——球碰撞后过于活跃，贴库球会沿库边高速滑动。经过多轮测试调整，最终确定为 0.88 / 0.77 / 0.99。期间 B 直接修改了 Common 层的 Constants.h 进行迭代，C 在后续整理中接受了这些参数变更。

\subsection{击球流程重构（A / C）}

初始设计为按钮触发击球（CueControl 点击→shootRequested→GameVM::shoot），但 A 在实现鼠标交互后，击球流程变更为：GameView 鼠标按下→球杆动画→鼠标松开→shotAnimationFinished→GameVM::shoot。这一变更导致 MainWindow 的信号连接需要重写（击球触发源从 CueControlVM 变为 GameView），同时角度/力度的同步从单向变为双向（鼠标在球桌上拖动和控件滑块需互相同步）。A 和 C 通过 Git 提交先后完成了 App 层的适配。

\section{总体心得}

五天的开发过程中，团队对 MVVM 架构的理解从课堂理论走向了工程实践。将程序拆分为五层的最大收益在于：三位开发者可以真正独立地并行工作，View 层在 Model 层尚未完整时即可通过 ViewModel 的骨架属性进行 UI 开发。但分层也带来了成本——跨层信号连接的维护、接口变更时的多方协调，都需要额外的时间投入。

从技术选型角度看，Qt 6 的信号/槽机制天然适配 MVVM 的数据绑定需求，Q\_PROPERTY + NOTIFY 信号组合可以简洁地实现 ViewModel 到 View 的属性推送。物理引擎方面，固定时间步长（1/60s）的设计保证了模拟的确定性，避免了帧率波动对碰撞检测精度的影响。

后续阶段的重点在于规则引擎的完善和测试覆盖——当前物理表现已经可用，但斯诺克复杂的规则体系（自由球、罚分、彩球复位等）是最具挑战性的部分。

\section{智能体的使用}

本项目选择\textbf{以人为主的开发模式}。大模型智能体在开发中仅用于补充性工作：代码补全与格式化、文档草稿生成、构建配置调试建议。所有核心架构设计（五层 MVVM 拆分方案）、接口协调（层间信号约定）、技术决策（vcpkg 取舍、物理参数选择）均由团队成员自主完成。

使用效果方面，智能体在重复性任务（如测试用例编写、文档格式化）上节省了时间，但对跨层逻辑的理解仍存在局限——MainWindow 的信号连接需要人工精确指定 lambda 中的属性同步逻辑，智能体难以自行推断完整的依赖关系。
