\section{技术要点}

\begin{itemize}
\item QPainter 抗锯齿渲染（Antialiasing + SmoothPixmapTransform）
\item QPropertyAnimation 实现球杆动画
\item 坐标系统分离：游戏世界坐标与屏幕像素坐标通过 gameToPixel 转换
\item View 层不持有 Model 指针，所有数据通过 GameViewModel::ballPositions 属性获取
\end{itemize}

\section{遇到的技术难点}

\subsection{坐标系统转换}

游戏世界使用以球桌中心为原点的坐标系（(-400,-200) 到 (400,200)），而 Qt 绘制使用像素坐标（左上角为原点）。初始实现时球的位置偏出屏幕，排查后发现 GameView 未能正确区分两种坐标系——从 ViewModel 获取的球位置可能是中心坐标系也可能是左上角坐标系。解决方案是在 drawBalls 中增加坐标系检测逻辑：遍历所有球，若任一球的 x/y 坐标超出 TABLE\_WIDTH/TABLE\_HEIGHT 范围，则判定为居中坐标系，通过 gameToPixel 的 centeredCoordinates 参数转换。

\subsection{球杆动画与鼠标交互的配合}

QPropertyAnimation 驱动球杆后拉-前推动画，但动画结束时需要立即触发真实物理击球。如果时序不当，用户可能在动画期间再次按下鼠标，导致二次击球。解决方案是：动画期间禁用鼠标按下事件处理，通过 shotAnimationFinished 信号确保动画-物理的串行衔接。

\section{个人心得与感悟}

View 层开发让我深刻体会到 MVVM 架构中“数据驱动界面”的含义——所有的绘制都来自 ViewModel 的属性变更通知，不再需要手动管理界面刷新。这种模式对调试非常友好：当球位置不对时，只需检查 GameViewModel::ballPositions 的值，问题要么在 Model 层（数据错误）要么在 View 层（绘制错误），而不会出现两层的交叉干扰。

鼠标交互的实现比预期复杂。从按钮击球切换到鼠标击球看似只换了输入源，但涉及角度计算、坐标转换、动画衔接、事件过滤等多个细节。这让我认识到 GUI 程序的交互设计需要充分考虑状态管理和时序控制。

与 C 的协作中，MainWindow 的信号连接经历了两次重构（按钮→鼠标、单向→双向同步），每次都需要在 App 层精确调整 lambda 逻辑。这让我理解了“集成层”的维护成本——越是灵活的解耦架构，中间层的胶水代码越需要细心维护。
